# Release Process

This project uses GitHub Releases as the changelog surface for action
consumers. Releases are tag-first so the tag-triggered workflow can run this
repository's own `release-notes-scribe` action and create a draft GitHub
Release before a maintainer publishes it.

## Required Local Checks

Run the standard validation loop before releasing:

```bash
npm run check
```

This runs TypeScript type-checking, Vitest, and the ncc bundle build. If
`dist/index.js` changes, commit the generated bundle with the release commit.

## Process

1. Choose the next semantic version tag, for example `v0.1.3`.
2. Run the release script:

```bash
npm run release -- v0.1.3
```

3. Optionally update the moving major tag in the same release:

```bash
npm run release -- v0.1.3 --move-major-tag
```

4. Wait for the `Draft Release Notes` GitHub Actions workflow to complete for
   the semantic version tag.
5. Review the drafted GitHub Release generated by this repository's local action
   through `uses: ./`.
6. Publish the GitHub Release when the notes are correct.
7. If `--move-major-tag` was used, confirm the moving major tag points at the
   same commit as the semantic version tag.

## Ordering Constraint

Do not create or publish the GitHub Release manually before the tag-triggered
`Draft Release Notes` workflow runs. The release notes are generated by
`.github/workflows/release.yml`, which checks out full history and runs this
repository's action with `uses: ./`.

The release script creates or updates local release artifacts, commits them when
needed, creates the tag, pushes the branch, and pushes the tag. The GitHub
Release is created afterward by the workflow as a draft. Maintainers publish the
draft after review.

The workflow only triggers on semantic tags matching `v*.*.*`. Moving major tags
such as `v0` are for consumers and should not create an additional release-note
draft.
